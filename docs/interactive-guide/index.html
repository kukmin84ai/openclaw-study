1 시스템 전체 구조 (System Architecture)
비유: 대형 우체국 — OpenClaw을 거대한 우체국에 비유해 봅시다. CLI는 정문, Gateway는 중앙 홀, 채널은 우편함, 에이전트는 배달부, 플러그인은 각 부서입니다. 아래 건물의 각 방을 클릭해 보세요!

왜 이렇게 만들었을까?

우체국처럼 각 부서가 독립적으로 일하면, 한 부서가 고장나도 다른 부서는 정상 작동합니다. OpenClaw도 같은 원리입니다 — CLI, Gateway, 채널, 에이전트가 각각 분리되어 있어서 Slack 채널에 문제가 생겨도 Discord 채널은 멀쩡하게 돌아갑니다. 이것을 "관심사의 분리(Separation of Concerns)"라고 합니다.

왜 이렇게 만들었을까?

처음부터 완벽한 시스템을 만들려고 하지 않았습니다. 작은 기능(511줄!)에서 시작해 사용자의 필요에 따라 점진적으로 성장했습니다. 이것이 소프트웨어 개발의 핵심 원칙인 "점진적 발전(Iterative Development)"입니다. 여러분도 프로젝트를 만들 때, 작게 시작하고 조금씩 키워 나가세요!

5 확장 생태계 (Extension Ecosystem)
비유: 앱스토어 — OpenClaw의 확장들은 스마트폰의 앱스토어와 같습니다. 소통 앱, 두뇌 앱, 도구 앱으로 나뉩니다. 각 버블을 클릭해 보세요!

📨 채널 확장 (소통 앱)

🧠 LLM 제공자 (두뇌 앱)

🔧 기능 확장 (도구 앱)

왜 이렇게 만들었을까?

스마트폰이 앱스토어 없이는 전화기일 뿐이듯, OpenClaw도 확장 없이는 빈 프레임워크입니다. 확장을 통해 누구나 새로운 기능을 추가할 수 있어서, 코어 팀이 모든 것을 만들 필요가 없습니다. 이것이 "플러그인 아키텍처"의 핵심 — 생태계가 스스로 성장합니다.

9 설계 원칙과 만드는 순서 (Design Principles & Build Order)
비유: 집 짓기 — 좋은 소프트웨어는 좋은 집처럼 설계 원칙이 필요합니다. 카드를 클릭하면 뒤집어서 설명을 볼 수 있습니다!
원칙 1
어댑터 패턴
클릭하여 뒤집기

채널과 프로바이더는 교체 가능한 어댑터입니다. USB-C 케이블처럼, 같은 모양의 플러그만 맞으면 어떤 기기든 연결됩니다. Slack을 빼고 Discord를 꽂아도 코어 코드는 한 줄도 안 바뀝니다.

원칙 2
플러그인 우선
클릭하여 뒤집기

내장 채널도 플러그인이고, 코어는 최대한 얇게 유지합니다. 특별한 코드가 아니라, 누구나 만들 수 있는 플러그인과 동일한 방식으로 내장 기능도 동작합니다.

원칙 3
훅 중심
클릭하여 뒤집기

25개 이상의 생명주기 훅으로 느슨한 결합을 만듭니다. 마치 이벤트 알림처럼, "메시지 도착 시", "응답 전" 등의 시점에 원하는 코드를 끼워 넣을 수 있습니다.

원칙 4
세션 격리
클릭하여 뒤집기

동시 사용자가 서로 간섭하지 않습니다. 카페에서 각자 테이블에 앉은 것처럼, A의 대화 내용이 B에게 보이지 않고, A의 오류가 B에게 영향을 주지 않습니다.

원칙 5
설정은 코드
클릭하여 뒤집기

JSON5 + Zod = 사람 친화적 + 타입 안전. 설정 파일이 주석도 쓸 수 있고(JSON5), 동시에 잘못된 값을 넣으면 즉시 알려줍니다(Zod 검증).

원칙 6
엄격한 경계
클릭하여 뒤집기

70개 이상의 SDK 서브패스로 교차 임포트를 방지합니다. 아파트의 현관문처럼, 허가된 입구만 사용해야 합니다. 이렇게 하면 코드가 서로 엉키지 않습니다.

원칙 7
이벤트 중심
클릭하여 뒤집기

모든 단계에 훅이 있는 메시지 파이프라인입니다. 공장의 컨베이어 벨트처럼, 메시지가 각 단계를 지나가면서 검사, 변환, 로깅이 자동으로 일어납니다.

🏗️ 만드는 순서 퀴즈

아래 7단계를 올바른 순서로 드래그하여 정렬해 보세요. 집을 지을 때 기초부터 쌓듯이, 소프트웨어도 순서가 있습니다!

왜 이렇게 만들었을까?

집을 지을 때 지붕부터 올리지 않듯, 소프트웨어도 순서가 중요합니다. 먼저 가장 단순한 기능(웹훅 릴레이)을 만들고, 그 위에 추상화(어댑터 인터페이스)를 쌓고, 마지막에 복잡한 기능(멀티 에이전트)을 추가합니다. 이것이 "바텀업 설계(Bottom-Up Design)"입니다. 여러분도 프로젝트를 만들 때, 가장 간단한 것부터 시작하세요!
