메시지 흐름 & 에이전트

메시지가 어떻게 여행하는지, 훅이 뭔지, 에이전트가 어떻게 동작하는지 알아봅시다.

레슨 2: 메시지 흐름 (Message Flow Pipeline)
비유: 식당 주문
손님(Slack)이 웨이터(채널 어댑터)에게 주문을 하면, 식당 매니저(Gateway)가 주문을 정리하고, 주방장(에이전트)이 요리하고, 다시 웨이터가 손님에게 배달합니다.

아래 파이프라인에서 "메시지 보내기" 버튼을 눌러 메시지가 흐르는 과정을 직접 확인해 보세요.

왜 이렇게 만들었을까?

메시지를 여러 단계로 나누면 각 단계를 독립적으로 테스트하고 교체할 수 있습니다. 예를 들어 Slack 대신 Discord를 쓰고 싶으면 "채널 어댑터"만 바꾸면 되고, 나머지는 그대로입니다. 이것을 관심사의 분리(Separation of Concerns)라고 부릅니다.

레슨 6: 훅 생명주기 (Hook Lifecycle)
비유: 지하철 노선도
각 역(hook)에서 승객(플러그인)이 타고 내립니다. 메인 노선을 따라가다가, 특정 역에서 갈라지는 지선(도구, 압축, 서브에이전트)도 있습니다.

역을 클릭하면 설명이 나옵니다. "열차 출발" 버튼으로 열차가 달리는 것을 확인하세요!

왜 이렇게 만들었을까?

훅 시스템 덕분에 OpenClaw의 핵심 코드를 수정하지 않고도 플러그인이 메시지 흐름의 원하는 시점에 끼어들 수 있습니다. 지하철 역에서 승객이 자유롭게 타고 내리듯, 플러그인도 필요한 훅에만 등록하면 됩니다.

레슨 7: 에이전트와 세션 (Agent & Session)
비유: 학교
교무실(라우팅)이 학생의 반을 배정하고, 학생증(세션 키)으로 신분을 확인하고, 담임선생님(에이전트)이 수업을 진행합니다.

1. 라우트 해석 (교무실)

메시지가 도착하면 "이 메시지를 어떤 에이전트가 처리할까?"를 결정합니다. 우선순위가 높은 바인딩부터 확인합니다.

2. 세션 키 구조 (학생증)

세션 키는 "이 대화가 어떤 맥락인지"를 나타냅니다. 4가지 패턴이 있습니다.

3. 에이전트 실행 (담임선생님)

에이전트는 LLM(AI 두뇌)을 호출하고, 필요하면 도구를 사용하고, 서브에이전트(보조교사)에게 일을 나눠줄 수 있습니다.

미니 퀴즈: 이 메시지는 어떤 세션으로?

왜 이렇게 만들었을까?

바인딩 우선순위 덕분에 "이 특정 사용자에게만 다른 에이전트를 연결"하는 것이 가능합니다. 학교에서 특정 학생에게 특별 담임을 배정하는 것처럼요. 세션 키 패턴이 다양한 이유는 DM, 그룹 채팅, 스레드 등 다양한 대화 형태를 모두 지원하기 위해서입니다.
