레슨 3: 플러그인 시스템 (Plugin Lifecycle)
비유: 레고 블록
플러그인은 레고 블록과 같아요.
발견(상자에서 블록 찾기) → 로딩(설명서 읽기) → 등록(조립판에 꽂기) → 실행(놀기!) 순서로 진행됩니다.
1
발견 (Discovery)

resolvePluginSourceRoots()가 3곳을 탐색합니다:

1. ~/.openclaw/extensions/ — 사용자가 직접 설치한 확장

2. 워크스페이스 node_modules/ — 프로젝트에 포함된 패키지

3. 내장 스톡 플러그인 — OpenClaw에 기본 포함

각 폴더에서 package.json에 openclaw 매니페스트가 있는지 확인하고, 발견하면 PluginCandidate[] 배열에 담습니다.

2
로딩 (Loading)

normalizePluginsConfig()로 활성/비활성 상태를 확인합니다.

jiti의 createJiti()로 동적 import — TypeScript, ESM, CJS 모두 지원!

결과물: OpenClawPluginModule 객체가 메모리에 올라갑니다.

3
등록 (Registration)

OpenClawPluginApi가 플러그인에게 조립판을 제공합니다:

registerChannel() registerProvider() registerHook() registerTool() registerHttpRoute() registerCommand()

각 함수를 호출하면 레고 블록이 조립판에 '찰칵!' 하고 꽂히는 것과 같아요.

4
실행 (Runtime)

createHookRunner(registry)가 run* 메서드들을 생성합니다.

훅은 우선순위(priority) 순서대로 실행돼요.

Gateway와 Agent들이 등록된 플러그인들을 사용하면서 메시지를 주고받습니다.

왜 이렇게 만들었을까?

발견 → 로딩 → 등록 → 실행 4단계로 나눈 이유는 느슨한 결합(loose coupling) 때문이에요. 각 단계가 독립적이라서:

- 새 플러그인을 추가해도 기존 코드를 고칠 필요가 없어요

- 특정 플러그인만 비활성화하기 쉬워요

- TypeScript든 JavaScript든 상관없이 로딩할 수 있어요 (jiti 덕분)

마치 레고처럼, 블록을 자유롭게 끼우고 빼는 것이 가능한 거예요!

레슨 8: 설정 파이프라인 (Configuration Pipeline)
비유: 요리 레시피
설정은 요리와 같아요!
재료(여러 소스에서 가져오기) → 계량(파싱) → 검수(검증) → 조리(최종 적용) 순서로 완성됩니다.
재료
Sources
계량
Parse
검수
Validate
조리
Output

3가지 재료(소스)가 우선순위 순서대로 섞입니다:

1. CLI 플래그 — 가장 높은 우선순위 (예: --port 3000)

2. 환경 변수 — 중간 우선순위 (예: OPENCLAW_PORT=3000)

3. JSON5 설정 파일 — 가장 낮은 우선순위

같은 설정이 여러 곳에 있으면, 높은 우선순위가 이깁니다!

재료를 정확히 계량하는 단계:

- JSON5.parse()로 설정 파일을 읽어요 (주석도 OK!)

- ${VAR} 형태의 변수를 실제 값으로 치환합니다

- includes로 다른 파일을 가져와 합칩니다

품질 검사 단계:

- 오래된 형식은 자동으로 새 형식으로 마이그레이션

- Zod 스키마로 타입과 값의 범위를 검사

- 플러그인 이름이 실제로 존재하는지 확인

검사 실패 시 친절한 에러 메시지와 함께 멈춥니다!

최종 완성 단계:

- 비어있는 값에 기본값 적용: agents, models, sessions, logging, compaction

- 결과물: OpenClawConfig (타입 안전, 검증 완료)

- Atomic write로 파일 저장 (중간에 꺼져도 안전)

- Backup rotation으로 이전 설정 자동 백업

직접 해보기: 설정값이 파이프라인을 통과하는 과정

소스를 클릭하면 값이 파이프라인을 통과합니다!

왜 이렇게 만들었을까?

재료 → 계량 → 검수 → 조리 파이프라인 구조의 장점:

- 우선순위 시스템: CLI > 환경변수 > 파일 순서 덕분에, 배포 환경마다 설정을 쉽게 바꿀 수 있어요

- Zod 검증: 잘못된 설정을 미리 잡아서 런타임 에러를 방지해요

- Atomic write: 설정 저장 중 프로그램이 꺼져도 파일이 깨지지 않아요 (임시 파일에 먼저 쓰고, 완료 후 이름을 바꿈)

- JSON5: 주석을 쓸 수 있어서 "이 설정이 왜 이 값인지" 메모할 수 있어요
